# 【6ヶ月完了】プロWebコンサルタント・ディレクター育成ロードマップ

**基本理念：「コードの実装はAIに任せ、粕谷は『戦略・対話・価値創造』に極振りする」**

このロードマップは、半年後に「クライアントの課題を解決し、的確なディレクションとコンサルティングができるプロフェッショナル」になるためのカリキュラムです。各Monthには**達成基準**を設定しており、チェックが全て埋まれば次のMonthに進むサインです。

> **前提**: 粕谷さんの役割は「ディレクター兼コンサルタント」です。コードを書く力ではなく、**クライアントの信頼を勝ち取り、プロジェクトを成功に導く力**を鍛えます。

---

## Month 1: 「共通言語」の獲得（Webのインフラと基本用語）

**目的**: クライアントやAIと自信を持って会話するための「用語と仕組み」をインストールする。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| インフラ用語 | ドメイン、サーバー、SSL、ホスティング、DNS |
| ファイル構造 | ディレクトリ構造、絶対パス/相対パス |
| 技術の概念 | HTML/CSSの役割（書き方は不要）、JavaScriptの役割 |
| 表示の仕組み | レスポンシブデザイン、ブラウザとキャッシュの関係 |

### AI活用学習法
1. **用語トーク練習**: 「〇〇ってクライアントに聞かれたら、どう説明すればいい？」と質問。例え話の引き出しを増やす。
2. **逆引きテスト**: 私（AI）が「サイトを公開したいんだけど、最初に何が必要？」のようなクライアント風の質問を出し、粕谷さんが回答する。
3. **教材**: `Webディレクター_20の基礎用語チートシート.md` を通読し、不明点を都度質問。

### 達成基準（Month 1 完了チェック）
- [ ] 20の基礎用語チートシートの全項目を、自分の言葉で説明できる
- [ ] 「ドメイン・サーバー・SSL」の関係を、家の例え話で30秒以内に説明できる
- [ ] クライアントに「キャッシュクリアしてください」と自信を持って案内できる
- [ ] HTML/CSS/JSの役割の違いを、一言ずつで言える

## Month 2: 「要件定義力」の向上（ヒアリングと制作フロー）

**目的**: プロジェクトを回すための「型」を身につける。設計に入る前に何を決めるべきかを体得する。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| 制作フロー全体像 | ヒアリング → 構成案（ワイヤーフレーム） → デザイントーン → 実装 → テスト → 公開・納品 |
| ヒアリング必須項目 | ターゲット（誰に）、目的（CV＝何をさせたいか）、競合、テイスト、必須要素、納期、予算 |
| 見積もりの考え方 | ページ数・機能・修正回数による工数の変動。相場観。 |
| 納品物の定義 | 何を渡すか（HTMLファイル一式？WordPress？）、保守契約の有無 |

### AI活用学習法
1. **ヒアリングロープレ**: 私（AI）が架空のクライアント（例：地元の歯医者、個人サロン、工務店など）を演じ、粕谷さんがヒアリングを行う。
2. **構成案レビュー**: 粕谷さんがヒアリング結果をもとに構成案を作成 → 私が「クライアント目線」と「エンジニア目線」の両方でフィードバック。
3. **見積もり相場リサーチ**: ココナラ・ランサーズ等で同種案件の価格帯を一緒にリサーチし、粕谷さんの適正価格を設定する。

### 達成基準（Month 2 完了チェック）
- [ ] ヒアリングシート（テンプレート）を1枚完成させている
- [ ] 架空案件でヒアリング → 構成案作成まで一人で完了できる
- [ ] 「LP 1ページ」「コーポレートサイト 5ページ」の概算見積もりを出せる
- [ ] 制作フローの6ステップを順番に言える

## Month 3: 「論理的デザイン」の理解（なぜこのUI/UXが良いのか）

**目的**: デザインの良し悪しを感覚ではなく「言葉と理由」で説明できる力を養う。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| 視線誘導 | Zの法則、Fの法則、グーテンベルグ・ダイアグラム |
| レイアウトの原則 | 近接・整列・反復・対比（デザインの4原則） |
| 余白と呼吸 | ホワイトスペースが「高級感」と「読みやすさ」を生む理由 |
| 色彩の基礎 | メインカラー・アクセントカラー・背景色の役割。色の心理効果。 |
| タイポグラフィ | フォントの種類（セリフ/サンセリフ/明朝/ゴシック）、行間と可読性 |
| モバイルUI | タップ領域の最小サイズ（44px）、親指ゾーン |
| Figmaの構造理解 | オートレイアウト（Flexboxの概念）、コンポーネント化、8pxグリッドルール |

### AI活用学習法
1. **サイト分析セッション**: 優れたWebサイトのURLを私に共有 → 「なぜ優れているか」を言語化して解説。
2. **ビフォーアフター比較**: 私がわざと「ダメなデザイン」と「良いデザイン」を並べ、違いを粕谷さんが言語化する練習。
3. **デザインフィードバック訓練**: 実際に制作中のサイトに対し、「ここの余白が〇〇な理由で狭い」のように具体的なフィードバックを言葉にする練習。
4. **Figmaマスタープランの実践**: 別途用意された `Webディレクター_Figma・デザインルールマスタープラン.md` に沿って、実装に耐えうる論理的なデザイン構造を学ぶ。

### 達成基準（Month 3 完了チェック）
- [ ] デザインの4原則（近接・整列・反復・対比）を実例つきで説明できる
- [ ] 参考サイトを見て、「このサイトが良い理由」を3つ以上言語化できる
- [ ] 色の選定理由を「心理効果」を交えてクライアントに提案できる
- [ ] スマホUIで「タップしづらいボタン」を指摘し、改善案を出せる
- [ ] Figmaで「オートレイアウト」を使ったボタン・カードを作成できる
- [ ] 余白を「8の倍数（8pxグリッドルール）」で設定する意味を説明できる

## Month 4: 「売るためのWeb（マーケティング）」の構築

**目的**: 綺麗なサイトではなく「売上・集客に直結するサイト」を設計・提案する視座を獲得する。Machi集客コンサルの中核能力。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| SEOの基本 | title / h1 / meta description の役割。表示速度と離脱率の関係。内部リンク構造。 |
| LPの構成理論 | FVの重要性 → 共感（課題提示）→ 解決策 → 権威性・実績 → ベネフィット → FAQ → CTA |
| コピーライティング | AIDA / PAS / 4P フレームワーク。ヘッドラインの書き方。 |
| CVR向上施策 | マイクロコピー、フォームの最適化（EFO）、CTAボタンの色・文言・配置 |
| アクセス解析 | Google Analytics 4 の基本画面（ユーザー数、セッション、直帰率）。Search Console での検索クエリ確認。 |

### AI活用学習法
1. **LP構成ディスカッション**: 「この商材を売るLP構成はどうすべきか？」を私と共同で設計する。
2. **コピーライティング添削**: 粕谷さんが書いたキャッチコピーやCTAテキストを、フレームワーク（AIDA等）に照らして私が添削・改善提案。
3. **競合サイト分析**: ライバル企業のLPを一緒に解剖し、「どこで勝てるか」を分析する。

### 達成基準（Month 4 完了チェック）
- [ ] LPの基本構成（7セクション）を順番に説明できる
- [ ] PASフレームワーク（Problem / Agitation / Solution）で30秒ピッチができる
- [ ] Google Analyticsの基本指標（ユーザー数・セッション・直帰率・CVR）を説明できる
- [ ] クライアントに「なぜこのCTAの文言にしたか」を論理的に説明できる

## Month 5: 「トラブルシューティング」と「AI連携」の極意

**目的**: 実務でのイレギュラー対応力と、AIへの指示精度を極限まで高める。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| 表示トラブルの一次切り分け | レイアウト崩れ → CSSの問題？ / 画像非表示 → パスの問題？ / 特定ブラウザのみ → 互換性の問題？ |
| デバイス差異 | PC・スマホ・タブレットでの表示差異。iOS Safari 特有の挙動。 |
| 公開後トラブル | 「修正が反映されない」→ キャッシュ / 「URLでアクセスできない」→ DNS浸透待ち / 「表示が遅い」→ 画像の最適化 |
| AIへの指示（プロンプティング） | 要件を「目的・制約条件・期待する出力形式」に分解して伝える技術 |
| スクリーンショットの活用 | 「画面のここがおかしい」を正確に伝えるためのスクショ＋赤枠＋テキスト注釈の習慣 |

### AI活用学習法
1. **トラブルシミュレーション**: 私がわざと壊れたページを作成 → 粕谷さんが症状を言語化 → 一次原因を推測してもらう。
2. **プロンプト改善道場**: 同じ要件を「曖昧な指示」と「精密な指示」で出し、出力の差を体感する。
3. **実案件の振り返り**: 過去の案件で発生した問題を体系化し、「トラブルシューティング辞書」を構築。

### 達成基準（Month 5 完了チェック）
- [ ] 「レイアウトが崩れた」と報告された時、3つの可能性を挙げられる
- [ ] クライアントからの「直ってない」に対し、キャッシュクリアの手順をデバイス別に案内できる
- [ ] AIへの指示を「目的・制約・出力形式」の3点セットで伝えられる
- [ ] スクリーンショット＋注釈で問題箇所を30秒以内に共有できる

## Month 6: 「コンサルティング（上流工程）」の確立

**目的**: 「言われたものを作る作業者」から、「経営課題をWebで解決するパートナー」へ完全に移行する。

### 学ぶべきこと
| カテゴリ | 項目 |
|---|---|
| 提案力 | クライアントの要望の「裏にある本当のニーズ」を見抜く力。「サイトが欲しい」の裏は「売上を上げたい」。 |
| クロスセル提案 | Webサイトだけでなく、LINE公式・MEO（Googleマップ対策）・SNS運用など周辺施策をセットで提案 |
| 継続収益モデル | 保守・更新契約、月次レポート提出、アクセス解析に基づく改善提案による月額顧問契約 |
| プロジェクト管理 | スケジュール管理、マイルストーン設定、クライアントへの進捗報告の仕方 |
| 専門家の振る舞い | 堂々と「できる・できない」を判断。根拠を持って代替案を提示。無理な値引きには毅然と対応。 |

### AI活用学習法
1. **提案書作成ワーク**: 架空のクライアント課題に対して、Webサイト＋周辺施策を含む「ソリューション提案書」を作成。私がレビュー。
2. **単価交渉ロープレ**: 値引き要求をしてくるクライアント（AI演技）に対して、価値を伝えて適正価格を守る交渉の練習。
3. **月次レポート作成**: Google Analyticsのデータを元に、改善提案を含む月次レポートのテンプレートを一緒に作成。

### 達成基準（Month 6 完了チェック）
- [ ] クライアントの「サイトを作りたい」を「売上を月〇万円上げたい」に変換してヒアリングできる
- [ ] Web制作＋LINE公式＋MEOのセット提案書を作成できる
- [ ] 月額保守契約の内容と金額を自信を持って提示できる
- [ ] 「それは対応範囲外です。代わりに〇〇をご提案します」と毅然と言える
