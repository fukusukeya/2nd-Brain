# Webディレクター必携：20の基礎用語＆概念チートシート（Month 1）

このチートシートは、Web制作のディレクションにおいて**「最も頻出する20の言葉」**を厳選したものです。

各用語には以下の4項目を統一フォーマットで記載しています。
1. **意味** — 正確な定義
2. **例え話** — クライアントに説明する際の分かりやすい比喩
3. **実務で出くわす場面** — いつこの知識が必要になるか
4. **ディレクターとして押さえるポイント** — 知っていると信頼度が上がる一歩先の知識

---

## 1. インフラ・環境編（インターネットの「土地と住所」）

### 1. ドメイン
* **意味**: インターネット上の「住所」にあたる文字列。例: `machistudio.com`
* **例え話**: 「Webサイトを『家』とするなら、ドメインは『住所』です。これがないと誰も辿り着けません。」
* **実務で出くわす場面**: 新規サイト制作時に「ドメインはお持ちですか？取得が必要ですか？」と確認するシーン。
* **押さえるポイント**: `.com` `.jp` `.co.jp` などの種類がある。`.co.jp` は法人のみ取得可能で信頼性が高い。取得はお名前.comやXserver等で年額1,000〜5,000円程度。

### 2. サーバー
* **意味**: Webサイトのデータ（HTML、画像等）を保管し、インターネットに公開するためのコンピューター。
* **例え話**: 「サイトという『家』を建てるための『土地』です。土地を借りないと家は建てられません。」
* **実務で出くわす場面**: 「サイトを公開したいのですが」と相談された際、ホスティング先の確認が必要。
* **押さえるポイント**: 月額1,000〜3,000円のレンタルサーバー（Xserver、さくら等）が個人〜中小企業の主流。大規模になるとAWS等のクラウドサーバーも選択肢に入る。

### 3. ホスティング
* **意味**: サーバーのスペースを貸し出すサービスの総称。レンタルサーバーやクラウドホスティングなど。
* **例え話**: 「土地（サーバー）を貸してくれる『不動産屋さん』のサービスです。」
* **実務で出くわす場面**: クライアントが「サーバーはどこがいいですか？」と聞いてくるシーン。
* **押さえるポイント**: 日本の中小企業案件ではXserverが圧倒的シェア。管理画面が日本語で、SSL設定やWordPressインストールがワンクリック。提案時に迷ったらXserverを推奨しておけば間違いない。

### 4. SSL化（https化）
* **意味**: サイト通信を暗号化する仕組み。設定完了後、URLが `http://` → `https://` に変わり、ブラウザに鍵マークが表示される。
* **例え話**: 「サイトの通信を暗号化して守る『セキュリティドア』です。未設定だとブラウザに『保護されていない通信』と赤字で警告が出てしまいます。」
* **実務で出くわす場面**: 納品前チェックリストの必須項目。未設定のまま公開すると、ユーザーの不安を煽り離脱に直結する。
* **押さえるポイント**: 無料SSL（Let's Encrypt）がXserver等で標準提供されている。新規公開時は必ず最初に設定する。SEOにも影響があり、Googleは「https」のサイトを優遇する。

### 5. DNS (Domain Name System)
* **意味**: ドメイン名（例: machistudio.com）を、サーバーの実際のIPアドレス（例: 123.45.67.89）に変換するシステム。
* **例え話**: 「『住所（ドメイン）』を入力すると、『実際の土地の座標（IPアドレス）』に案内してくれる『カーナビ』です。」
* **実務で出くわす場面**: サーバーを引っ越す際にDNS設定の変更（ネームサーバーの切り替え）が必要。設定変更後、反映まで最大72時間かかることがある。
* **押さえるポイント**: DNS変更後に「まだ新しいサイトが表示されない」とクライアントに言われたら、「DNSの浸透（反映待ち）です。通常数時間〜最大72時間で切り替わります」と落ち着いて案内する。

## 2. ファイル・構造編（Webサイトの「骨組みと装飾」）

### 6. HTML (HyperText Markup Language)
* **意味**: Webページの構造（見出し・本文・画像・リンクなど）を定義する言語。全てのWebページの骨格。
* **例え話**: 「家の『骨組みと壁』です。どこに柱を立て、どこに窓をつけるかを決める設計図の役割です。」
* **実務で出くわす場面**: クライアントから「テキストを修正してほしい」と言われた際、変更箇所はHTMLファイル内にある。
* **押さえるポイント**: HTMLの「タグ」を全て暗記する必要はない。ただし `<h1>`（大見出し）、`<p>`（段落）、`<a>`（リンク）、`<img>`（画像）の4つだけは「見て何か分かる」程度に認識しておくと、AIへの指示精度が上がる。

### 7. CSS (Cascading Style Sheets)
* **意味**: HTMLの見た目（色、フォント、サイズ、余白、レイアウト）を制御する言語。
* **例え話**: 「家の『壁紙・インテリア・外装ペイント』です。骨組み（HTML）は同じでも、CSSを変えれば和風にもモダンにもなります。」
* **実務で出くわす場面**: 「もう少し文字を大きくしてほしい」「色を変えたい」といった見た目の修正は全てCSSで対応。
* **押さえるポイント**: 「デザイン修正＝CSSの修正」と覚えておく。HTMLの構造自体を変える必要がない修正（色・大きさ・間隔の変更）は比較的軽い作業であることが多い。

### 8. JavaScript (JS)
* **意味**: サイトに「動き」や「インタラクション」をつける言語。ポップアップ、スライダー、フェードイン、フォームの入力チェック等。
* **例え話**: 「自動ドアやセンサーライトのように、ユーザーの操作に反応して動く『仕掛け』を作る役割です。」
* **実務で出くわす場面**: 「スクロールしたら画像がふわっと出てくるようにしたい」「スライダーで画像を自動で切り替えたい」などの要望。
* **押さえるポイント**: 動きのある機能（アニメーション、ポップアップ等）は工数が少し増える。クライアントから「簡単に動きをつけてほしい」と言われた場合でも、仕様を明確にする必要がある（速度、タイミング、トリガー等）。

### 9. ディレクトリ構造
* **意味**: サーバー（もしくはPC）内でのフォルダとファイルの階層構造。
* **例え話**: 「家の中の『部屋の間取り図』です。リビングにテレビ、キッチンに冷蔵庫があるように、画像は `/img` フォルダ、CSSは `/css` フォルダに入れます。」
* **実務で出くわす場面**: AIに修正を指示する際、「どのファイルのどこを直すか」を正確に伝える場面。
* **押さえるポイント**: 一般的なWebサイトの構成は以下の通り。この構造が頭に入っていると、ファイルを探す時間が激減する。
  ```
  /プロジェクト名/
  ├── index.html       ← トップページ
  ├── css/style.css    ← デザイン定義
  ├── js/main.js       ← 動きの定義
  └── img/             ← 画像フォルダ
  ```

### 10. 絶対パス / 相対パス
* **意味**: ファイルの場所を指定する2つの方法。絶対パスは `https://example.com/img/photo.jpg` のようにフルURL、相対パスは `../img/photo.jpg` のように現在の場所からの道順。
* **例え話**: 絶対パス＝「東京都渋谷区〇〇1-2-3」（どこからでも辿り着ける完全な住所）。相対パス＝「この道をまっすぐ行って2つ目の角を右」（今いる場所からの道順）。
* **実務で出くわす場面**: 「画像が表示されない」「リンクが切れている」といったトラブルの原因の大半がパスのミス。
* **押さえるポイント**: トラブル報告を受けた際は、まず「PCでは見えるか？スマホだけか？全デバイスか？」を確認し、次に「いつから？（何か変更した後か？）」を確認する。この2つの質問でパスの問題かどうかの見当がつく。

## 3. デバイス・表示編（ユーザーからの見え方）

### 11. レスポンシブデザイン
* **意味**: 画面サイズ（スマホ・PC・タブレット）に合わせて、レイアウトが自動で切り替わるデザイン手法。
* **例え話**: 「折りたたみ式の家具のように、部屋の広さ（画面サイズ）に合わせて形を変える仕組みです。」
* **実務で出くわす場面**: ほぼ全ての案件で必須。提案時に「レスポンシブ対応込みです」と伝える場面。
* **押さえるポイント**: 2024年時点で日本のWebトラフィックの約75%がスマホ経由。「スマホファースト（スマホの表示を最優先で設計する）」が現在の業界標準。見積もり時にPC/スマホ両対応が標準であることを明記し、追加料金にしないほうが信頼感が高い。

### 12. UI (User Interface / ユーザーインターフェース)
* **意味**: ユーザーの目に触れ、操作する部分の総称。ボタン、メニュー、フォーム、フォント、配色など。
* **例え話**: 「レストランの『メニュー表や店内装飾』です。見やすく、選びやすく、美しいことで満足度が上がります。」
* **実務で出くわす場面**: デザイン確認時に「このUIだとボタンが目立たない」「フォームが使いにくい」などのフィードバックをする場面。
* **押さえるポイント**: UIの良し悪しは「ユーザーが迷わず目的を達成できるか」で評価する。個人の好みではなく、「ターゲットユーザーにとって」使いやすいかが判断基準。

### 13. UX (User Experience / ユーザーエクスペリエンス)
* **意味**: サイトを通じてユーザーが得る総合的な「体験」。使いやすさ、心地よさ、スムーズさを含む。
* **例え話**: 「レストラン全体の『接客・雰囲気・味・お会計のスムーズさ』です。UIが見た目だとしたら、UXはそこで過ごした時間の満足度そのもの。」
* **実務で出くわす場面**: クライアントに提案する際、「綺麗なだけでなく、お客様がスムーズに問い合わせまで辿り着ける設計をします」と伝える場面。
* **押さえるポイント**: UI（見た目）が良くてもUX（体験）が悪い例 →「デザインはおしゃれだけど、問い合わせボタンがどこにあるか分からない」。UXを意識した設計ができることが、プロとアマチュアの最大の差別化ポイント。

### 14. ファーストビュー (FV / First View)
* **意味**: サイトを開いた瞬間に画面に表示される領域。スクロールせずに見える範囲。
* **例え話**: 「お店の『看板とショーウィンドウ』です。通りがかった人（訪問者）がここで3秒以内に興味を持たなければ、店に入らず（スクロールせず）立ち去ります。」
* **実務で出くわす場面**: LP・コーポレートサイト問わず、最もデザインと文言に時間をかけるべき領域。
* **押さえるポイント**: FVに入れるべき3要素 → ①キャッチコピー（何のサイトか一目で伝わる）②メインビジュアル（信頼性・世界観を表現）③CTA or ナビゲーション（次のアクションへの導線）。この3つが揃っているかを確認するだけで、FVの品質判断ができる。

### 15. キャッシュ
* **意味**: ブラウザが「過去に開いたサイトのデータ」を端末内に一時保存し、次回アクセス時に高速表示する仕組み。
* **例え話**: 「一度読んだ本のページに付箋を貼っておくようなもの。次に開く時は早いが、本の内容が改訂されても古い付箋を見てしまうことがある。」
* **実務で出くわす場面**: 修正を反映したのにクライアントから「変わってないんですが…」と連絡が来るシーン。最も頻出するトラブル。
* **押さえるポイント**: 対処法を即座に案内できると信頼度が大幅UP。
  * **PC（Chrome）の場合**: `Ctrl + Shift + R`（Mac: `Cmd + Shift + R`）でスーパーリロード
  * **スマホ（iPhone Safari）の場合**: 設定 → Safari → 履歴とWebサイトデータを消去
  * **スマホ（Android Chrome）の場合**: Chrome → 設定 → プライバシーとセキュリティ → 閲覧履歴データの削除

## 4. マーケティング・運用編（サイトの「売上と集客」）

### 16. SEO (Search Engine Optimization / 検索エンジン最適化)
* **意味**: Googleなどの検索エンジンで、特定のキーワードで上位に表示されるようにサイトを最適化する施策。
* **例え話**: 「Googleという巨大な商店街で、お店を目立つ場所（1ページ目）に出店するための対策です。」
* **実務で出くわす場面**: クライアントからの「Googleで検索して出てくるようにしてほしい」という要望への対応。
* **押さえるポイント**: SEOには「内部対策（サイト構造・タグの最適化）」と「外部対策（被リンク獲得等）」がある。Web制作で担うのは主に内部対策。成果が出るまで3〜6ヶ月かかるのが通常で、「来月には1位になります」とは絶対に言わないこと。

### 17. CV / CVR (コンバージョン / コンバージョン率)
* **意味**: CV＝サイト上で達成したい目標行動（購入、問い合わせ、LINE登録、資料請求など）。CVR＝訪問者のうちCVに至った割合（%）。
* **例え話**: 「CVは『ゴール（成約）』、CVRは『100人来店して何人がゴールしたかの確率』です。」
* **実務で出くわす場面**: 提案時に必ず定義する。「このサイトのCV（ゴール）は何ですか？」はヒアリングの最重要質問の一つ。
* **押さえるポイント**: 業界平均CVRの目安（参考値）
  * LP（ランディングページ）: 2〜5%
  * ECサイト: 1〜3%
  * BtoBコーポレートサイト: 0.5〜2%
  * この数字を知っていると、「CVRが1%なら100人来て1人が問い合わせる計算です」のように具体的な提案ができる。

### 18. LP (Landing Page / ランディングページ)
* **意味**: 特定の目的（商品販売、問い合わせ獲得など）に特化した、1枚の縦長ページ。
* **例え話**: 「会社のパンフレット（コーポレートサイト）が百貨店なら、LPは『1つの商品だけを全力で売る凄腕の実演販売員』です。」
* **実務で出くわす場面**: 「とりあえずサイトが欲しい」というクライアントに「まずはLP1枚から始めませんか？」と提案するシーン。
* **押さえるポイント**: LPの標準構成（7セクション）を覚えておくと提案がスムーズ。
  1. ファーストビュー（キャッチコピー＋ビジュアル＋CTA）
  2. 共感・課題提示（「こんなお悩みありませんか？」）
  3. 解決策の提示（サービス紹介）
  4. 権威性・実績（数字、メディア掲載、資格等）
  5. お客様の声（社会的証明）
  6. よくある質問（FAQ）
  7. CTA（最終アクション誘導）

### 19. CMS (Content Management System / コンテンツ管理システム)
* **意味**: プログラミング知識がなくても、管理画面からサイトのコンテンツ（ブログ、お知らせ等）を追加・編集できるシステム。代表例はWordPress。
* **例え話**: 「ブログや『お知らせ』を、メールを書くような感覚で更新できる『管理画面つきのサイト』です。」
* **実務で出くわす場面**: クライアントが「自分でブログを更新したい」「お知らせを随時追加したい」と言った時。
* **押さえるポイント**: CMSの導入＝制作工数が増える（静的HTML制作より高い）。導入の要否はクライアントの更新頻度で判断する。月1回程度の更新なら、こちらで代行する保守契約の方が安上がりになるケースも多い。WordPress以外にもSTUDIO、Wixなどのノーコードツールも選択肢。

### 20. Google Analytics (GA4) / Search Console (GSC)
* **意味**: Googleが無料提供するアクセス解析ツール群。GA4はサイト訪問者の行動分析、GSCは検索キーワードやインデックス状況の確認に使う。
* **例え話**: 「GA4は店内の『防犯カメラ兼来客カウンター』（誰が来て何をしたか）。GSCは『通行量調査データ』（どんな言葉で検索して来たか）です。」
* **実務で出くわす場面**: サイト公開後の効果測定。「公開したけど効果が出てるのか分からない」に答えるためのツール。
* **押さえるポイント**: サイト公開時に必ず両方を設定する（設定はAIに任せてOK）。クライアントへの月次レポートで使う基本指標は以下の4つ。
  * **ユーザー数**: 何人がサイトを訪れたか
  * **セッション数**: 何回サイトが見られたか（1人が2回来れば2セッション）
  * **直帰率**: 1ページだけ見て離脱した割合（高いと改善が必要）
  * **コンバージョン数/率**: 目標（問い合わせ等）の達成数と達成率
